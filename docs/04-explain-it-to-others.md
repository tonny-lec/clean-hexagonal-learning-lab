# 04. 最終的に人に説明できるレベルにするための整理

ここでは、他人に説明するときの言い方まで落とし込みます。

---

## 30秒版の説明

> Clean / Hexagonal Architecture は、業務の中心ロジックをフレームワークや DB、外部 API などの詳細から分離する設計です。  
> 中核は port を通じて外界とやり取りし、adapter がその具体実装を担います。  
> その結果、変更に強く、テストしやすくなります。

---

## 1分版の説明

> この設計は、業務ルールをアプリの中心に置き、依存方向を内側へ保つ考え方です。  
> Use Case は「何をしたいか」を表し、Domain は「何が正しいか」を表します。  
> DB や外部 API は port / adapter の境界の外側に置きます。  
> だから UI や DB が変わっても、中核ロジックを守りやすく、ユースケース単位のテストも高速に回せます。

---

## 3分版の説明

### 1. なぜ必要か
現場では、Controller や ORM に業務ルールが入り込みやすく、変更やテストがつらくなります。

### 2. 何をするのか
そこで、
- Domain
- Use Case
- Port
- Adapter

に役割を分けます。

### 3. 何がうれしいか
- 中核ロジックを守れる
- 外部依存を差し替えやすい
- テストしやすい
- 入口が増えても再利用しやすい

### 4. いつ使うか
- 複雑な業務ルールがあるとき
- 長期運用するプロダクト
- 外部連携が多いとき
- 複数の入口から同じ処理を使うとき

### 5. いつ使わないか
- 単純 CRUD
- すぐ捨てる試作
- 学習コストに見合わない小さな機能

---

## 用語を一言で言えるかチェック

- **Domain**: 業務そのものの知識
- **Entity**: 業務上重要な概念
- **Use Case**: 何をしたいか
- **Port**: 外に何を依頼するかの抽象
- **Adapter**: Port の具体実装
- **Dependency Rule**: 依存を内側へ向ける原則

---

## サンプルコードを使った説明例

このプロジェクトでは、注文作成を題材にしています。

- `Order` は業務ルールを持つ
- `placeOrder` は注文作成フローを表す
- 価格取得、保存、支払いは port にする
- in-memory repository や console payment は adapter
- `index.ts` で実装を組み立てる

つまり、

> 注文するという業務の中心は内側にあり、DB や決済の詳細は外側にある

ということをコードで見せています。

## 自分の言葉で説明するテンプレート

最後は、暗記ではなく自分の言葉で説明できるのが大事です。
このテンプレートを使うと話しやすいです。

> この設計は、まず業務の中心を切り出して守るためのものです。  
> 変わりやすい UI、DB、外部 API から中核ロジックを分離します。  
> そのために Use Case と Domain を内側に置き、外部依存は Port と Adapter で扱います。  
> 複雑な業務や長期運用のシステムでは有効ですが、単純な CRUD ではやりすぎになることもあります。
