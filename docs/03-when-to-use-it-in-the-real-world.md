# 03. 現場ではどう役立つのか、どの場面で使うのか、どう判断するのか

ここでは、"概念として分かる" から一歩進めて、"現場でどう使うか" に落とし込みます。

---

## どんな価値があるのか

## 1. 変更に強くなる
たとえば次の変更が来ても、中核ロジックを守りやすくなります。

- ORM の変更
- DB の変更
- 決済サービスの変更
- REST API から GraphQL への変更
- 管理画面追加
- バッチ追加

理由は、外部都合を port / adapter の境界に押し出しているからです。

---

## 2. テストしやすくなる
Use Case を、
- 本物の DB なし
- 本物の HTTP なし
- 本物の決済なし

で確認できます。

これは現場でかなり大きいです。

- PR の確認が速い
- リファクタリングしやすい
- 失敗時の原因が追いやすい

---

## 3. 責務分離しやすい
コードレビューで、次のように会話しやすくなります。

- これは domain に置くべきか？
- これは use case の責務か？
- これは adapter に閉じるべきか？

責務の会話がしやすい設計は、チーム開発で強いです。

---

## よくある利用シーン

### シーン1: 外部連携が多い業務システム
例:
- 決済
- 在庫
- メール送信
- 認証基盤
- SaaS 連携

こういう場面では、外部変更の影響を局所化しやすいです。

### シーン2: 複数インターフェースを持つシステム
例:
- 同じ業務処理を API とバッチの両方から使う
- 管理画面とバックエンドジョブで同じユースケースを使う

このとき、中核を再利用しやすいです。

### シーン3: 業務ルールが増えていくシステム
例:
- 割引条件が複雑
- 承認フローがある
- 例外条件が増える

ルールが Controller に散るより、中心に寄せたほうが保守しやすいです。

---

## 採用判断の目安

### 採用を強く検討する
- 1年以上運用する見込み
- 業務ロジックがすでに複雑、または複雑化しそう
- DB や SaaS を差し替える可能性がある
- テスト速度が重要
- 同じ業務を複数経路から呼ぶ

### 部分採用で十分
- まだ小さいが、ユースケースだけ分けておきたい
- repository だけ抽象化したい
- 一部の外部依存だけ境界を切りたい

### 見送ってよい
- 単純 CRUD の管理画面
- すぐ捨てる試作品
- 学習コストより素早い実装が重要

---

## 現実的な判断フレーム

次の 4 点で見ると実務で判断しやすいです。

## 1. Core は何か？
まず、守りたい中心があるかを確認します。

例:
- 注文ルール
- 価格計算
- 承認ロジック
- 配送可否判定

これが薄いなら、重い設計は不要です。

## 2. Detail は何か？
差し替わりやすい技術詳細を洗い出します。

例:
- DB
- Web フレームワーク
- Queue
- 外部 API
- 認証サービス

## 3. 変化の向きはどこか？
どちらが先に変わるかを考えます。

- UI は変わる
- API 仕様は変わる
- SaaS は変わる
- でも業務ルールは残る

なら、中心を守る価値があります。

## 4. テスト戦略に効くか？
ユースケースを速く回せると、開発速度そのものが上がります。

---

## アンチパターン

### 1. なんでも interface 化する
変わらないものまで抽象化すると、読みづらくなるだけです。

### 2. Domain が薄く、Use Case が巨大
Domain に置くべきルールが Use Case に流出している可能性があります。

### 3. Adapter に業務判断が混ざる
たとえば controller 側で重要判断をしていると、再利用性が落ちます。

### 4. ルールより構造を守ることが目的になる
「このフォルダに置くべき」ばかり気にして、
- 何を守りたいのか
- なぜ切るのか

が見えなくなると本末転倒です。

---

## 現場向けの説明テンプレート

採用理由をチームに説明するときは、次の型が使えます。

> この機能は、業務ルールが今後増えやすく、さらに外部依存の変更も起こりそうです。  
> そのため、ユースケースとドメインを中核に置き、DB や外部 API はポート越しに切り出します。  
> こうすると、テストを速くしつつ、将来の差し替えコストを下げられます。

逆に採用しない説明もできます。

> 今回は単純 CRUD が中心で、複雑な業務ルールも複数入口もありません。  
> そのため、全面的な Clean / Hexagonal は採用せず、Use Case 分離だけに留めます。

この「採用する理由」「採用しない理由」を言語化できることが重要です。

---

## 学習者向けチェックリスト

自分に次を問いかけて、説明できれば理解が進んでいます。

- このシステムの core は何か？
- external detail は何か？
- どの変更から何を守りたいのか？
- port にする価値があるのは何か？
- adapter に閉じるべき都合は何か？
- 今回はフル採用か、部分採用か、非採用か？

---

## 一言まとめ

現場での価値は、

> **「きれいに見えること」ではなく、「変更とテストに強くなること」**

です。

この観点で判断できると、Clean / Hexagonal を道具として使えるようになります。
