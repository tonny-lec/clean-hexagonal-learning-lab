# 01. そもそも何が課題になるときに使うのか

Clean / Hexagonal アーキテクチャは、**コードをきれいに並べるための流行語** ではありません。
本質は、**ビジネスの中心ロジックを外部都合から守るための設計** です。

---

## まず、何がつらいのか

現場でよく起きるつらさは、だいたい次のどれかです。

### 1. 業務ルールが UI やフレームワークの中に埋まる
例:
- Controller に条件分岐が増える
- ORM の Entity に業務判断が混ざる
- React コンポーネントの中で重要な業務計算をしている

この状態の問題:
- ルールの所在が分からない
- 再利用できない
- テストしづらい
- 画面変更が業務ルール変更のように見えてしまう

### 2. DB や外部 API の都合に設計が引っ張られる
例:
- DB のテーブル形状がそのまま業務モデルになっている
- 外部決済 API のレスポンスに内部ロジックが依存している
- 「今はこの SaaS を使っているから」という理由で中核が固定される

この状態の問題:
- 外部サービスを替えにくい
- テストでモック祭りになる
- 仕様変更時の影響範囲が広い

### 3. テストが遅い・重い・壊れやすい
例:
- ちょっとした業務ロジック確認に DB が必要
- API サーバーを起動しないと確認できない
- テストが E2E に偏り、失敗原因が追いにくい

この状態の問題:
- 変更に慎重になりすぎる
- 小さく直せない
- リファクタリングが止まる

### 4. 入口が増える
例:
- 同じ業務処理を API / CLI / Batch / Message Consumer から使いたい
- 管理画面と公開 API の両方から同じユースケースを呼びたい

この状態の問題:
- 同じ処理が複製される
- 一方だけ仕様がズレる
- バグ修正が片方にしか反映されない

---

## Clean / Hexagonal が解く中心課題

一言でいえば、

> **業務の中心を、外側の技術詳細から分離すること**

です。

ここでいう「外側の技術詳細」には、たとえば次が含まれます。

- Web フレームワーク
- DB / ORM
- 外部 API
- メッセージキュー
- CLI
- UI
- ログ・監視・通知

これらは重要ですが、**ビジネスの本質そのものではない** はずです。

---

## どんなチーム・システムで効きやすいか

### 向いている状況
- プロダクト寿命が長い
- 業務ルールが複雑
- 複数の外部システムと連携する
- テスト容易性が重要
- 将来の変更が多そう
- チーム開発で責務分離が必要

### 向いていない状況
- 数日で捨てる検証コード
- 単純 CRUD だけで十分
- 業務ルールが薄く、ほぼデータ入出力しかない
- 学習コストに見合うほどの複雑さがない

---

## 誤解しやすいポイント

### 誤解1: レイヤーを増やせばよい
違います。
大事なのは**依存方向の制御**です。
ファイルを増やしただけでは価値は出ません。

### 誤解2: すべてを抽象化すべき
違います。
変わる可能性・切り替えたい可能性が低いものまで全部 port にすると、ただの儀式になります。

### 誤解3: とにかく疎結合にすれば正義
違います。
疎結合にはコストがあります。
理解負荷、実装量、初期開発速度を下げる可能性があります。

---

## 現場っぽい判断のしかた

次の質問に "はい" が多いほど、採用価値があります。

- この業務ルールは数年残りそうか？
- UI や DB を変えても残る中心知識があるか？
- 複数の入口から同じ処理を呼ぶか？
- DB / SaaS / メッセージ基盤を差し替える可能性があるか？
- テストを高速に回したいか？
- Controller や ORM にルールが漏れ始めているか？

---

## この学習ラボで扱う課題

このプロジェクトでは、注文作成を題材にして次を分離しています。

- 注文という業務概念
- 注文作成ユースケース
- 価格取得
- 支払い
- 保存

ここで、
- 価格取得
- 支払い
- 保存

は外部依存です。

一方、
- 注文をどう組み立てるか
- 合計金額をどう計算するか
- 空の注文を許すか

は中核ロジックです。

この切り分け感覚が、Clean / Hexagonal 理解の入口です。

---

## 一言まとめ

Clean / Hexagonal は、

> **「変更されやすい外側」と「守りたい内側」を分ける設計」**

です。

この視点を持てると、単なるディレクトリ構成の話ではなく、
「なぜこの設計を選ぶのか」を説明できるようになります。
