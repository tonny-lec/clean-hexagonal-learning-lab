# 08. 設計者のように考え、最終的に人に説明できる状態にする

この章は、知識を「説明力」と「設計判断力」に変えるための整理です。

---

## まず持つべき視点

Clean / Hexagonal を使うときに見るべきは、フォルダ名ではありません。
見るべきは次の 4 つです。

1. **何が core か**
2. **何が detail か**
3. **どこが変わりやすいか**
4. **どこに責務を閉じ込めるべきか**

---

## このラボを設計者目線で読むと

### Core
- 注文を成立させるルール
- 金額計算
- 参照結果の意味

### Detail
- HTTP body
- CLI text
- batch input
- SQLite 保存方法
- console payment の実装

### 境界
- request parsing
- DTO mapping
- port implementation
- error to response mapping

### 判断の要点
- 内側を detail に引っ張られないようにする
- detail の変更を局所化する
- テストを速くする

---

## 複数 entrypoint があるときの説明

このラボでは同じ use case を:
- CLI
- Batch
- HTTP

から呼べるようにしています。

ここで伝えたいことは、

> **入口が増えても、業務フローそのものは増やさない**

という設計感覚です。

つまり、入口ごとにロジックを複製するのではなく、
入口ごとの都合だけ adapter に閉じ込めます。

---

## 「全面採用か部分採用か」をどう話すか

### このラボのように全面的に学ぶ価値があるとき
- 長期運用する
- 複数の外部依存がある
- 複数 entrypoint がある
- 業務ルールが増える

### 部分採用でよいとき
- use case だけ分ける
- repository だけ port 化する
- HTTP request を domain へ直結しない

### 見送ってよいとき
- 単純 CRUD
- 一時的ツール
- 学習コストに合わない小規模機能

---

## 5分で説明するテンプレート

> この設計は、業務の中心を外部都合から守るためのものです。  
> 中核には Domain と Use Case を置き、外部依存は Port と Adapter で扱います。  
> その結果、UI や DB、外部 API が変わっても中核ロジックを守りやすくなり、テストも高速に回せます。  
> ただし、何にでも全面採用するわけではなく、業務ルール・寿命・変更頻度・複数入口の有無を見て判断します。

---

## チームに説明するときの観点

### 1. 何を守りたいのか
- 注文ルール
- 金額計算
- 業務フロー

### 2. 何が変わりそうか
- 決済サービス
- 保存方法
- API 形式
- 入口の数

### 3. そのためにどこを切るのか
- port
- adapter
- DTO
- presenter
- repository

### 4. そのコストに見合うか
- 学習コスト
- 実装量
- チームの理解度

## 自分が本当に理解できたかを測る質問

次に答えられれば、かなり理解できています。

1. Clean と Hexagonal の違いは？
2. なぜ port を置くのか？
3. request DTO と domain model を分ける理由は？
4. presenter は何を守るのか？
5. idempotency を考えないと何が起きるか？
6. query を command と分ける価値は？
7. in-memory と SQLite adapter の違いは何を学ばせるか？
8. この案件でフル採用 / 部分採用 / 非採用のどれを選ぶか？

---

## 最後のゴール

最終ゴールは、

> **この構造を覚えることではなく、課題に対して理由付きで設計を選び、その理由を他人に説明できること**

です。

このラボを読み終えたら、次はぜひ自分で次をやってみてください。

- 別の題材で use case を 1 本追加する
- その機能に HTTP adapter を足す
- どこを port にして、どこを直実装にするか自分で決める
- その判断理由を 3 分で説明する

それができれば、理解はかなり実戦レベルに近づいています。
